# 개발 및 수정 프로세스 기록 (Implementation & Modification Processes)

본 문서는 인천소방학교 이전 신축공사 내역서 대시보드 개발 과정에서 발생한 주요 단계, 시행착오, 오류 해결 및 스타일 변경 사항을 상세히 기록합니다.

---

## 1. 초기 개발 단계 (Initial Setup)

### 1.1 데이터 파이프라인 구축
- **작업 내용**: 4개 공종(건축, 기계, 조경, 토목)의 엑셀 파일을 파이썬(Pandas)으로 분석하여 SQLite 데이터베이스로 통합.
- **시행착오**: 각 엑셀 파일마다 데이터 시작 행(skiprows)과 컬럼 구조가 달라 개별적인 파싱 로직을 구현해야 했음.

### 1.2 표준 분류 체계 도입
- **변화**: 단순한 DB 구조에서 국가 건설공사 표준분류체계(20개 컬럼) 규격으로 전면 개편.
- **기계설비 분리**: 기계내역서 하나에 섞여 있던 기계, 전기, 소방 항목을 품명 키워드 기반으로 자동 분류하는 알고리즘 적용.

---

## 2. 디자인 및 UI/UX 변천사 (Design Evolution)

### 2.1 1단계: 다크 모드 & 글래스모피즘
- **스타일**: 세련된 다크 테마와 반투명한 유리 효과 적용.
- **피드백**: 가독성이 낮고 데이터 테이블이 좁아 실무용으로 불편하다는 지적.

### 2.2 2단계: 틸트 시프트 & 미니어처 스타일
- **실험적 시도**: "장난감 미니어처" 컨셉의 디자인 가이드라인에 따라 AI 배경 이미지와 상하단 블러(Tilt-shift) 효과 적용.
- **수정 사항**: 원색 위주의 강조색과 둥근 폰트를 사용하여 장난감 패키지 같은 느낌 구현.
- **결과**: 시각적으로 독특했으나 실무 데이터 분석 도구로서는 너무 화려하다는 판단하에 스타일 변경 결정.

### 2.3 3단계: 실무형 프로페셔널 스타일 (최종)
- **최종 확정**: 슬레이트(Slate) 블루 톤의 깔끔하고 정돈된 비즈니스 스타일로 개편.
- **강점**: 넓은 여백, 명확한 타이포그래피, 눈이 편안한 색상 배합으로 실무 데이터 작업에 최적화.

## 3. 주요 기능 추가 (Feature Enhancements)

### 3.1 건물 동(Building) 분류 및 필터링
- **필요성**: 전체 내역을 건물별(본관동, 관사동, 훈련탑 등)로 나누어 보고자 함.
- **구현**: ETL 로직에서 `WHERE2_동` 컬럼에 키워드 매핑 기능을 추가하여 데이터베이스 재구축. 프론트엔드에 '건물 동' 전용 필터 탭 신설.

### 3.2 CSV 데이터 내보내기
- **기능**: 대시보드에서 조회 중인 원본 데이터를 실무용 엑셀(CSV)로 다운로드하는 기능 추가.
- **최적화**: 한글 깨짐 방지를 위해 UTF-8 BOM 처리를 백엔드 스트리밍 시 적용.

## 4. 배포 및 오류 해결 (Deployment & Troubleshooting)

### 4.1 Vercel 배포 404 에러
- **현상**: GitHub 푸시 후 Vercel 접속 시 페이지를 찾을 수 없는 현상 발생.
- **원인**: 백엔드와 프론트엔드가 섞인 모노레포 구조에서 Vercel의 루트 경로 인식 오류.
- **해결**: `vercel.json` 설정 파일을 작성하여 프론트엔드 빌드 경로와 백엔드 API 경로를 수동으로 지정 및 라우팅 설정.

### 4.2 데이터 표시 누락 (0건 표시)
- **현상**: 배포된 사이트에서 "데이터가 0건"으로 나타남.
- **원인**: 
  1. `.gitignore` 설정으로 인해 SQLite DB 파일(`project_costs.db`)이 GitHub에 업로드되지 않음.
  2. Vercel 서버리스 함수 실행 시 상위 폴더의 파일 접근 경로 문제.
- **해결**: 
  1. 데이터베이스 파일을 `backend/` 폴더 안으로 이동.
  2. 모든 소스 코드의 DB 경로를 상대 경로(`os.path.dirname(__file__)`)로 통일.
  3. gitignore를 수정하여 DB 파일을 강제로 포함시켜 푸시.
